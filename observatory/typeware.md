Software is fascinating; becuase it's inherently non-physical, but it's interacting with both; physical & symbolic elements (visual, audio, interaction, etc.)



What's interesting to me is how systems-software is designed...

- Operating-systems and Kernels

- Runtimes

- Drivers

- Firmware (incredible stuff)

- Engines, Editors



And there's one common principle that makes all of these systems cohesive, from an engineering perspective; architectural continuity.



An operating system has:

- Semaphores

- Memory models

- Algorithms



A much smaller program; such as a website, feels almost...



This principle of continuity in systems-software, may explain why

- coding a game engine instead of a game, contains inherent tangibility

- coding a game without an engine, contains near limitless possibility.



A game engine has standardized modules, features, models, and general expectations fo usefluness. Most of this stuff can be built from first-principles engineering, reverse engineering other engines, and studying material online. That's essentially how 3D modelling engines such as Blender, were developed (sourcing from the public, recorded history of it's development.)



However, designing a game contains little to none of these systems. 



If this applies to games, then ocnsequentially, it may also apply to many (if not, all) forms of software engineering. And in today's information age; the technology we interface with everyday affects and shapes us; this matters way beyond code itself; but how we use code as a language to build technologies that shape human life.



Essentially, I am studying the phenomenology of software design; what makes the difference between benecifial technologies, and wasteful (or even harmful) technologies?



That's the core theme.
