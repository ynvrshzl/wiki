# "Cybernetica" 

A Cybernetic programming interface between machine and human. An executable grammar and classifcation codex of universal computational laws, systems, architectural patterns, structures and concepts for today's information age.

## 5.0.0 - Raw stream of thoughts...
The core tension of programming; **Expression vs. Constraint.**

We see this polarity in synonymous forms;

- Structure vs. Freedom
- Finite vs. Non-finite
- Order vs. Chaos
- Linearity vs. Non-linearity
- Complexity vs. Simplicity
- Pattern vs. Novelty
- Verbosity vs. Abstraction\*

This implies the issue is multi-factorial, from a common ancestor; information density vs. information surface area


### 5.0.1 - "Verbosity vs. Abstraction"

\*This is the closest tension to the fundamental argument in this study; **Expression vs. Constraint** is arguably a surface-level property of the fundamental balance of all information-based tools; the balance of **Verbosity vs. Abstraction.**



Arguably, these properties are what code both; powerful as an expression tool of immediate thought, but volatile as a systems medium.



If we approach software today as a longterm architecture engineering project, it is clear that the range of expression in code is a volatile factor.


### 5.0.2 - The universal software engineering issue.
Most commonly, I find this recurring issue... most engineers may also commonly go through; 

- Designing systems is hard. A system relies on muscle-memory, time, state and long-term use. These are all **temporal metrics of the immesureable factor; human cognition resource as currency.**
- We either design the system before the code, or the code before the sytsem
- These must be fundamentally different modes of operation, becuase they require different cognitive resources.

- We end up getting lost in abstraction. In designing constraints without foundations. In creating APIs, models, DSLs, without...

## Design
The primary design of Cybernetica is based on this hypothesis: "code today is objective, what if code described function instead?"

e.g., instead of *"function \<name\> (<params\>)"* we describe *"Role(\<args\>).verb(\<supporting-args\>)"*

## Abstract
Cybernetica is a family of solutions in this area. Specifically aiming to solve the issue, at the substrate-level; the code itself.

## Hypothesis
Hypothesis *"if the baseline of software engineering, is architecturally-fluent, will this free human cognitive resources from systems design, to systems engineering?"*

## 0.0 - Proposition
*An introductory proposition by Hazl Lzah, the author of this project:* 

Today in code, there are too many ways to do the same thing. This is a natural artifact, an artifact of a field where *execution comes first, design is last.* All complex fields &mdash; from Mathematics to Engineering &mdash; at one point in their history, exhibited this phenomenon.

"When there are too many options, there is no standard. Without standards, there are no foundations for the systems we build. In an interconnected world, the standard is the fortitude of communication. Without structure, systems decay and collapse."

- What human software engineers inevitably end up building (classical universal systems)

### 0.0.3 - Attention economics

*"A wealth of information, creates a poverty of attention." [^1]*

This quote speaks volumes as to the current issues of systems design in modern software engineering; **there are too many ways to do the same thing.** 

We need to design tools as if **Human cognitive resources** are the fundamental currency of energy and information exchange in a system; becuase this is arguably exactly what computational systems become. A **Cybernetic** exchange of information and energy, where language is the fundamental tool for both **authoring and understanding** computer systems.

Traditionally, complex fields have separated these domains becuase they are fundamentally different. They require different modes of human cognition. Code is ambigious by nature, which is a natural consequence of all technological phenomena.[^2]

#### 0.0.4 - Phenomenology
When we have too many choices and not enough standards, this creates an issue of information overload, especially when making choices in real-time;

*e.g., "Should this be an object, or a function? Should this variable be camel case, or snake case? Should I wrap this in an interface?"*

That may sound trivial, but over time, these micro-decisions waste limited human cognitive resources. These kind of repetitive design choices should be standard to an interface, language, framework or tool; not a visible agent.

#### 0.0.5 - The paradox of choice

Statistically, there are more ways to do things incorrectly, than correctly. This is why serious fields have developed battle-tested foundations at the structural level. They say; 

*"We will not work in these domains, until we can design systems according to risk."*
