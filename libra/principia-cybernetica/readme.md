## Continuity vs. Fragmentation 
Arguably the biggest problem when authoring systems is the tension between **Continuity and Fragmentation.**

Fragmentation is not inherently negative, and Continuity is not always helpful; these elements require **context.**

### Cognitive Fragmentation
In the **context** of authoring code, we are often dealing with **fragmented attention:** x files, y functions, z dependencies, w references, etc. These are all properties of a system; yet most of it is never inspectable. We **cognitively simulate** the system as we author it, and when we can no longer reason about the code itself, we rely on output to inspect the system. When the output itself becomes plauged with errors, this leads to **Cognitive Fragmentation.**

The problem with fragmentation, is that while it is useful to break down large problems, eventually we human beings cannot navigate space without continuity, it cannot be fragmented. this is why we string things together. we interpolate. it is about orientation and navigation. here is an illustrated example; 

- 3 files
- 12 functions 
- 24 variables

Imagine we have no way of querying, or analyzing the structure of this code. We are simply cognitively simulating code as we read and write it. this is creating a nonlinear structure, and other fields have solved this by visualizing, chunking, and standardizing. the problem in coach today is that these structures are authorable, so we never have the opportunity to switch from; 
- authoring mode

## Structure of code

Today, we may think of code as being composed entirely of primitives. loops, functions, variables, etc. the problem is, unlike logic gates or mathematics, programming allows us to author, co-create with the language. we don't currently have tools to support this process, because it is almost entirely phenomenological; it is implicit. 

the problem is that, while in Logic gates, the engineer is strictly limited by hardware, just like the early computer scientist pioneers, they had to work with strict constraints and limitations. today, we can create essentially anything. people have created game engines, databases, spreadsheets, flight simulators, task management apps, calendars, phone books, operating systems, and so much more digital systems. these digital systems are becoming integrated into real parts of our human lives. do we start to depend on them because they start to hold information, and give us access to information, that we would not otherwise have biologically. that is the cybernetic significance of Technology today. 

the problem is that code is mechanically definable, we can literally author the substrate. we can create our own rules. we can design our own little language, from within a larger generalized language. eventually the systems we build outgrow the language capabilities, and we start defining our own systems, our own rules, our own systems. the problem is, to arrive at this stage we cannot merely architect and design the system from the beginning, this almost always leads to over engineering. we also cannot merely sample and experiment. 

today, we can define a class, a function, a string, a sequence, as both a functional operator, and a mechanical device; we can define new types, we can define logic and order and structure for how these parts will communicate. this is how a system starts to emerge, this is why the ability to author code, is both its greatest strength, and it's most volatile property.
